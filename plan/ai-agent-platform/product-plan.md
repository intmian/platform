# AI / Agent Product Plan

## 1. 背景

当前仓库中的 AI 能力更接近“单次模型调用”的封装，主要用于一问一答、改写、总结、翻译等轻量能力。

后续目标不是只维护一个调用 SDK 的工具层，而是逐步形成一个可扩展的 AI 能力产品体系：

1. OpenAI、Copilot 等模型能力提供方会成为统一的能力来源。
2. 上层不再只是一问一答，而是会逐步演进成不同能力等级的 Agent。
3. Agent 未来会支持设定注入、不同模型等级、工具链、计划能力、记忆能力，以及不同场景下的裁剪版本。

当前需求还没有完全收敛，因此本计划先以产品方向和阶段目标为主，不提前锁定底层实现方案。

## 2. 产品目标

这个方向最终希望形成一个统一的 AI 能力产品层，满足以下目标：

1. 对内提供稳定、统一、可组合的 AI 能力入口。
2. 同时支持“最简单的单次问答”和“更复杂的多步骤 Agent”。
3. 让模型、能力、权限、记忆、工具都可以按场景裁剪，而不是做成单一重型 Agent。
4. 让 OpenAI、Copilot 等能力来源在产品层表现一致，避免业务直接依赖具体厂商。
5. 支持后续逐步产品化，而不是一次性做完整闭环。

## 3. 产品定位

这个方向的产品定位不是单一聊天机器人，而是一个分层演进的 AI 能力平台。

短期定位：

1. 统一模型调用能力。
2. 提供最小可用的问答 Agent。
3. 支持按任务选择不同模型档位。

中期定位：

1. 提供可配置的任务型 Agent。
2. 支持工具调用与多步执行。
3. 形成可复用的 Agent preset。

长期定位：

1. 支持具备计划能力的复杂 Agent。
2. 支持带记忆和长期行为设定的 Agent。
3. 支持更丰富的能力编排和多 Agent 协作可能性。

## 4. 典型产品形态

后续不应只存在一种 AI 产品形态，而应该至少支持以下几类：

1. 裸调用能力
   面向最简单的文本生成、改写、总结、翻译、结构化输出等场景。
2. 单次问答 Agent
   看起来像 Agent，但本质是一次运行、一次输出、无长期状态。
3. 轻量任务 Agent
   支持传入设定、模型选择、上下文和有限工具。
4. 工具型 Agent
   可以调用工具完成任务，但不一定有复杂计划能力。
5. 计划型 Agent
   可以围绕目标拆分步骤、执行、修正、继续推进。
6. 记忆型 Agent
   在任务执行之外，还会保存偏好、上下文、长期信息。

## 5. 目标用户与使用场景

当前先按内部产品能力规划，不限定最终只服务某一个业务。

预期用户包括：

1. 业务服务层
   需要直接调用 AI 完成简单任务。
2. 内部平台能力层
   需要构建可复用 Agent 能力。
3. 后续具体业务场景
   例如改写、总结、问答、分析、自动执行、辅助操作等。

优先支持的场景梯度：

1. 简单文本任务
2. 带设定的单次问答
3. 带工具的任务执行
4. 带步骤推进的复杂任务
5. 带记忆和个性化设定的长期 Agent

## 6. 产品演进路线

### 阶段 A：统一模型能力入口

目标：

1. 先把 OpenAI、Copilot 这样的能力来源统一为平台可消费的基础能力。
2. 让业务使用方不再直接理解具体厂商差异。
3. 保持当前简单场景可继续使用。

阶段结果：

1. 单次调用能力稳定可用。
2. 支持按场景切换模型档位。
3. 支持后续接更多 provider。

这一阶段不追求：

1. 复杂 Agent。
2. 长期记忆。
3. 高级计划系统。

### 阶段 B：最小 Agent 化

目标：

1. 把当前一问一答的能力升级为“最简单可运行的 Agent”。
2. 引入设定、上下文、模型控制、输出风格控制等产品能力。
3. 形成可被不同业务复用的基础 Agent 形态。

阶段结果：

1. 可以创建不同配置的基础 Agent。
2. 可以按任务创建 mini Agent、normal Agent 等不同档位。
3. 可以支持单次运行的标准化输入输出。

### 阶段 C：工具型 Agent

目标：

1. 让 Agent 不只是回答问题，而是具备调用工具完成任务的能力。
2. 能在“模型推理”和“工具执行”之间形成基本闭环。
3. 开始支持更强的任务场景。

阶段结果：

1. 有限工具调用能力上线。
2. 能根据任务类型使用不同工具权限。
3. 可以形成一批标准工具型 Agent。

### 阶段 D：计划型 Agent

目标：

1. 让 Agent 能围绕目标做分步推进。
2. 支持任务拆解、执行、修正、重试、终止。
3. 让复杂任务不再依赖外部手写流程。

阶段结果：

1. 可以处理多步骤任务。
2. 计划与执行形成闭环。
3. 开始具备“代理执行”的雏形。

### 阶段 E：记忆与长期能力

目标：

1. 让 Agent 能保留用户偏好、场景背景和任务历史。
2. 支持个性化和持续性行为。
3. 为长期 Agent 或复杂协作场景打基础。

阶段结果：

1. 支持可控记忆。
2. 支持不同记忆级别的 Agent。
3. 开始具备长期行为一致性。

## 7. 产品能力清单

未来能力可以按以下维度逐步扩展：

1. Provider 选择
   OpenAI、Copilot，后续可继续扩展。
2. 模型选择
   支持按任务指定模型，支持 mini、normal、advanced 等分档。
3. Agent 设定
   支持传入角色设定、风格设定、任务约束。
4. 上下文管理
   支持单次上下文和多轮上下文。
5. 工具能力
   支持外部工具、内部工具、受限工具。
6. 执行模式
   支持单次问答、循环执行、分步执行。
7. 计划能力
   支持目标拆分和步骤管理。
8. 记忆能力
   支持短期记忆、偏好记忆、长期记忆。
9. 能力裁剪
   支持创建只有 mini 模型、无工具、无记忆的轻量 Agent。
10. 安全与权限
   支持对模型、工具、记忆能力进行权限控制。

## 8. 产品策略

### 8.1 模型策略

模型不应写死在能力实现中，而应成为产品配置的一部分。

产品侧需要支持：

1. 按任务选择模型。
2. 按 Agent 类型绑定默认模型。
3. 允许创建只具备 mini 模型能力的轻量 Agent。
4. 允许高阶 Agent 使用更强模型。
5. 在成本、速度、质量之间做可配置权衡。

### 8.2 Agent 策略

不是所有 Agent 都要具备所有能力。

产品侧建议长期保留以下差异化路线：

1. 裸 Agent
   只做最基础能力，成本最低。
2. 标准问答 Agent
   适合通用文本交互。
3. 工具 Agent
   适合执行型任务。
4. 计划 Agent
   适合复杂任务推进。
5. 记忆 Agent
   适合长期交互和个性化能力。

### 8.3 迭代策略

这个方向要避免一开始就做成大而全平台。

建议的产品迭代原则：

1. 先统一能力入口，再做 Agent。
2. 先有最小 Agent，再做复杂 Agent。
3. 先支持工具，再做计划。
4. 先验证短期有效场景，再决定是否投入长期记忆。
5. 每个阶段都保留可单独交付的产品成果。

## 9. 阶段性里程碑

### 里程碑 1：形成统一 AI 能力产品入口

完成标志：

1. 业务可以不感知具体 provider 差异。
2. 模型选择不再散落在业务代码中。
3. 简单任务可以稳定使用统一入口。

### 里程碑 2：推出最小问答 Agent

完成标志：

1. 一问一答能力以 Agent 形式运行。
2. 支持传入设定和模型档位。
3. 支持创建轻量 Agent。

### 里程碑 3：推出工具型 Agent

完成标志：

1. Agent 可以调用工具完成任务。
2. 不同 Agent 可以有不同工具权限。
3. 工具能力开始可复用。

### 里程碑 4：推出计划型 Agent

完成标志：

1. Agent 可以执行多步任务。
2. 复杂任务不再依赖单轮问答。
3. 有基本任务推进能力。

### 里程碑 5：推出记忆型 Agent

完成标志：

1. Agent 可以保存和读取可控记忆。
2. 长期行为开始具备一致性。
3. 可针对不同业务选择不同记忆级别。

## 10. 当前明确的非目标

当前这版计划先不做以下承诺：

1. 不承诺一次性完成完整 Agent 平台。
2. 不承诺当前就确定最终架构。
3. 不承诺所有 Agent 都支持记忆和计划。
4. 不承诺所有 provider 在第一阶段具备完全一致能力。
5. 不承诺现在就细化到任务级排期。

## 11. 当前需要持续观察的问题

这些问题现在先记录，不在本轮强行定案：

1. Copilot 在产品中更适合作为通用模型源，还是更适合作为特定能力源。
2. Agent 的记忆是否需要分层，例如偏好、任务、长期知识。
3. 工具能力的开放边界如何定义。
4. 高级 Agent 是否需要和普通 Agent 保持统一产品入口。
5. 后续是否需要支持多 Agent 协作。
6. 模型选择是否需要强约束模板，还是允许完全外部传入。

## 12. 下一步建议

本计划之后的文档细化顺序建议如下：

1. 先拆“场景规划”，明确先服务哪些任务类型。
2. 再拆“版本路线图”，明确先交付哪些产品形态。
3. 再拆“能力矩阵”，明确不同 Agent 的能力差异。
4. 最后再进入技术设计和实现拆分。
