# Current-Architecture Agent Plan

## 1. 目的、需求

### 1.1 目的

本计划聚焦一个明确目标：

在不依赖 `cc-connect` 作为核心框架的前提下，完全基于当前平台既有架构，落地一个可长期演进的 Agent 系统。

这里的“当前架构”指仓库现有的后端平台能力，包括：

1. 单进程服务容器。
2. 平台级配置中心。
3. SQLite 存储能力。
4. HTTP / RPC 网关。
5. 定时任务与后台轮询能力。
6. 统一 AI 配置和模型调用封装。

本方案的基本判断是：

1. 当前架构可以承载 Agent 的核心能力。
2. `cc-connect` 如果未来要接入，只应该作为渠道层或适配层，而不应承担人格、记忆、主动性和上下文编排的核心职责。
3. 这一轮应优先把 Agent 核心内核做在平台内，再决定是否接微信、Web 或其他外部入口。

### 1.2 需求范围

目标 Agent 不是一次性问答工具，而是具备持续交互能力的长期 Agent。核心需求包括：

1. 有稳定人设。
2. 有多轮上下文能力，而不是每轮独立回答。
3. 有长期记忆，能提取和复用用户偏好、习惯、重要事件。
4. 能根据聊天状态判断是否回复、如何回复、是否延迟回复。
5. 能在受控条件下主动开启话题或跟进。
6. 能访问实时信息，例如天气、新闻、地点和外部事件。
7. 能隐藏工具调用和内部编排过程，保留自然聊天体验。
8. 能在平台内长期演进，而不是被某个接入框架绑定。

### 1.3 本轮非目标

为了控制范围，本计划暂不把以下内容作为首轮强目标：

1. 不在本轮锁定微信接入方案。
2. 不把 `cc-connect` 作为系统中心。
3. 不要求第一阶段就具备复杂多 Agent 协作。
4. 不要求第一阶段就支持完整 Tool Calling 协议。
5. 不要求第一阶段就达到全自动、高频主动聊天。

### 1.4 成功标准

如果第一阶段交付完成，至少应满足以下判断标准：

1. 平台内可以创建并运行一个独立 Agent 服务。
2. Agent 可以接收外部消息并基于历史上下文生成回复。
3. Agent 可以把高价值信息沉淀为长期记忆。
4. Agent 可以基于规则和状态决定是否回复。
5. Agent 可以在必要时调用实时信息能力。
6. 后续无论接微信、Web 还是其他渠道，都不需要重写核心编排逻辑。

## 2. 方案、架构

### 2.1 总体方案

建议在现有后端服务体系内新增一个独立的 `agent` 服务，而不是把逻辑塞进现有 `auto`、`cmd` 或未来的接入层。

推荐形态：

`platform + agent service + agent.db + realtime adapters + channel adapters`

其中：

1. `platform` 负责服务注册、共享配置、共享存储、鉴权、HTTP / RPC 接入和生命周期管理。
2. `agent service` 负责会话编排，是整个系统的核心。
3. `agent.db` 负责消息、主题摘要、长期记忆、主动任务和缓存数据。
4. `realtime adapters` 负责天气、新闻、地点等外部信息获取。
5. `channel adapters` 只负责消息收发，不负责记忆和人格。

### 2.2 核心分层

建议把 Agent 系统拆成以下几层：

1. 接入层
   接收外部消息，统一转成平台内部的消息输入。
2. 编排层
   负责上下文整理、记忆检索、决策、工具调用判断和最终回复生成。
3. 记忆层
   负责长期记忆、主题摘要、会话状态和主动任务持久化。
4. 实时信息层
   负责外部实时数据获取、缓存和失败重试。
5. 生成层
   负责调用模型生成最终自然语言输出。

### 2.3 服务内模块划分

建议新增目录结构：

1. `backend/services/agent/service.go`
   服务注册、启动、停止、RPC 分发。
2. `backend/services/agent/msg.go`
   输入输出协议定义。
3. `backend/services/agent/orchestrator/`
   会话主流程、决策器、上下文构造器。
4. `backend/services/agent/store/`
   SQLite 访问、DDL、仓储接口。
5. `backend/services/agent/memory/`
   记忆抽取、打分、去重、召回。
6. `backend/services/agent/topic/`
   主题切分、摘要压缩、历史块重组。
7. `backend/services/agent/tools/`
   天气、新闻、搜索、时间等实时信息适配器。
8. `backend/services/agent/scheduler/`
   主动消息任务、延迟回复、后台扫描。
9. `backend/services/agent/prompt/`
   人设模板、决策模板、摘要模板。

### 2.4 存储设计

建议新增独立数据库 `agent.db`，避免把复杂会话结构全部塞进共享键值存储。

建议最少包含以下表：

1. `sessions`
   保存会话主信息和渠道绑定信息。
2. `messages`
   保存原始消息和必要的处理标记。
3. `topic_segments`
   保存主题级摘要块。
4. `memories`
   保存长期记忆。
5. `conversation_state`
   保存关系状态、互动节奏和回复风格提示。
6. `jobs`
   保存延迟任务、主动消息任务和待执行触发器。
7. `outbox`
   保存待发送消息，便于未来接不同渠道。
8. `tool_cache`
   保存天气、新闻等短期缓存。

### 2.5 消息处理主流程

每次收到消息后，不应直接进入模型，而应走统一编排流程：

1. 写入 `messages`。
2. 读取最近原始消息窗口。
3. 召回相关主题摘要。
4. 召回相关长期记忆。
5. 读取当前 `conversation_state`。
6. 判断是否需要实时信息。
7. 运行决策器，得到动作类型。
8. 如果需要回复，再组装 prompt 调用模型。
9. 写入回复结果、状态更新、记忆候选和后续任务。

建议决策器至少支持以下动作：

1. `reply_full`
2. `reply_brief`
3. `ask_followup`
4. `ack_only`
5. `wait_for_more`
6. `silent`
7. `propose_topic`

### 2.6 上下文与记忆策略

上下文建议做双层设计：

1. 近程上下文
   始终保留最近一段原始消息，保证语气和当前话题不断裂。
2. 中远程上下文
   更早消息按主题压缩为摘要块，而不是简单截断。

长期记忆与主题摘要要分层：

1. 主题摘要回答“之前聊过什么”。
2. 长期记忆回答“这个人长期稳定的信息是什么”。

长期记忆应带结构化字段，例如：

1. `title`
2. `category`
3. `tags`
4. `content`
5. `confidence`
6. `source`
7. `status`

### 2.7 主动能力策略

主动性必须受控，不应默认高频触发。

建议只允许以下三类主动触发：

1. 记忆触发
   例如用户的重要日期、长期偏好、之前承接过的话题。
2. 实时事件触发
   例如天气变化、用户关注主题的重大事件。
3. 会话延续触发
   例如上一轮用户明确表达兴趣点，适合稍后自然续接。

主动机制需要频控：

1. 限制同一会话的最小主动间隔。
2. 限制单日主动次数。
3. 用户近期明显冷淡时暂停主动。
4. 深夜等敏感时段不主动。

### 2.8 与现有平台的结合方式

当前架构下，Agent 服务应优先复用平台已有能力：

1. 复用平台服务注册机制。
2. 复用共享 `CfgExt` 作为人设、模型、工具策略配置入口。
3. 复用平台 AI 配置读取逻辑。
4. 复用定时与后台轮询模式处理主动任务和缓存刷新。
5. 复用 HTTP / RPC 网关作为调试和外部接入入口。

但以下部分不建议直接复用现有实现，而应独立建设：

1. 不把聊天消息存进 `xstorage` 的普通键值结构。
2. 不把会话主动性强行并入 `auto` 服务的日报逻辑。
3. 不把 Agent 核心逻辑做成某个渠道插件内部逻辑。

### 2.9 分阶段实施建议

#### 阶段 1：最小可运行 Agent

目标：

1. 独立 `agent` 服务上线。
2. 固定人设可配置。
3. 最近消息窗口可用。
4. 长期记忆表可用。
5. 基础回复决策器可用。
6. 可接最少一类实时信息能力。

#### 阶段 2：上下文压缩与主动能力

目标：

1. 主题切分与历史摘要上线。
2. 记忆抽取打分与去重上线。
3. `jobs` 和 `outbox` 机制上线。
4. 延迟回复和低频主动跟进上线。

#### 阶段 3：渠道接入与体验增强

目标：

1. 接入微信或其他渠道。
2. 打通外部收发和内部 Agent 内核。
3. 加强主动性策略、实时信息策略和关系状态管理。

## 3. 已实现部分和缺口

### 3.1 当前已实现部分

基于现有代码，平台已经具备以下可复用基础：

1. 平台级服务容器
   已支持统一注册、启动和 RPC 分发。
2. 共享存储和配置中心
   已有平台级 SQLite 存储和 `CfgExt`。
3. AI 配置与模型档位
   已支持统一读取模型配置、场景到模型档位映射。
4. 基础 AI 调用封装
   已支持基于系统提示词和用户输入发起模型调用。
5. 定时任务能力
   已有基于 cron 的任务单元机制。
6. 后台轮询能力
   已有平台内定时巡检与状态更新先例。
7. HTTP / RPC 入口
   已可用于 Agent 调试入口、管理入口和后续适配层接入。

这些能力意味着：

1. 当前平台已经具备承载 Agent 的基础运行环境。
2. 当前平台并不缺“宿主”，缺的是“Agent 编排内核”。

### 3.2 当前缺口

要实现目标 Agent，当前最主要的缺口包括：

1. 缺少独立的 `agent` 服务。
2. 缺少会话级编排器。
3. 缺少消息、摘要、记忆、任务的专用数据模型。
4. 缺少多轮上下文构造能力。
5. 缺少主题切分与压缩机制。
6. 缺少长期记忆抽取、评分、去重、召回机制。
7. 缺少回复决策器。
8. 缺少主动任务调度模型。
9. 缺少 `outbox` 这一层，无法优雅支持多渠道发送。
10. 缺少面向实时信息的专用工具适配层。

### 3.3 当前实现与目标之间的关键差异

现有 AI 能力更接近“单次任务型调用”，而目标 Agent 需要“长期状态型调用”。

差异主要体现在：

1. 现有调用方式是单轮输入输出，目标需要多轮会话管理。
2. 现有任务偏日报、改写、总结，目标需要持续关系和行为状态。
3. 现有定时能力偏静态任务，目标需要会话级、动态任务调度。
4. 现有数据存储偏配置和结果缓存，目标需要结构化会话与记忆存储。

### 3.4 首批需要补齐的能力

如果进入开发，建议先补齐以下缺口：

1. `agent` 服务骨架。
2. `agent.db` 和基础仓储层。
3. 入站消息协议和最小会话模型。
4. 最近消息窗口和记忆召回。
5. 回复决策器 MVP。
6. 实时信息适配器 MVP。
7. 人设模板和生成模板。

### 3.5 风险与约束

实施时需要注意以下约束：

1. 当前 AI 调用封装更适合单轮 prompt，后续可能需要增强为多消息接口。
2. 主动性如果没有频控和状态约束，容易变成打扰。
3. 记忆如果没有评分和去重机制，会很快被污染。
4. 如果把渠道接入和核心内核一起做，会放大复杂度。
5. 如果继续复用面向静态任务的结构来承载会话级状态，后续维护成本会很高。

### 3.6 当前结论

结论如下：

1. 在当前平台架构下实现该 Agent 是可行的。
2. 最合理的做法是新增一个独立 `agent` 服务。
3. `cc-connect` 即使未来接入，也应处于外围适配层，而不是系统中心。
4. 这项工作的关键不是再找一个接入框架，而是把会话编排、记忆分层和主动性机制设计清楚。
