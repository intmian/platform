**目标**
这个 agent 需要同时满足这些能力：

- 有稳定人设，和你说话的方式、语气、偏好保持一致
- 隐藏工具调用和思考过程，聊天体验自然
- 能读取聊天记录，避免上下文断裂
- 不只是被动回答，还会判断是否该回复、是否该延迟、是否该主动开启话题
- 有长期记忆，能把重要信息整理后存入 SQLite
- 记忆不是散乱文本，而是带 `tag`、`title`、`category` 的结构化知识
- 能联网查实时信息，比如天气、时事、地点信息
- 对重大事件有一定“主动知晓”能力，而不是只能被问到才查
- 在长期互动中维持关系感、连续性和话题延展能力

**框架定位**
`cc-connect` 适合做这套系统的“外壳层”，不适合单独承担“人格、记忆、判断”全部逻辑。

它负责：

- 微信个人号接入
- 消息收发
- session 管理
- 定时任务
- 底层 agent 调用

真正的核心要额外加一层“会话编排内核”：

- 人设控制
- 记忆存储与检索
- 上下文压缩
- 回复决策
- 主动对话策略
- 实时信息接入

所以最终不是“只配 `cc-connect`”，而是：

`cc-connect + conversation orchestrator + sqlite memory + realtime tools + agent`

**整体架构**
建议拆成 6 层。

1. 接入层
   `cc-connect + weixin`
   负责从微信收消息、把回复发回去。

2. 编排层
   `conversation orchestrator`
   这是主控层。每次收到消息，先不直接回复，而是统一做上下文整理、记忆检索、联网判断、回复决策。

3. 记忆层
   `SQLite`
   存长期记忆、主题摘要、对话状态、消息索引。

4. 检索层
   负责从 SQLite 中拉出：
   - 最近消息
   - 历史主题摘要
   - 相关长期记忆
   - 用户偏好和关系状态

5. 实时信息层
   负责天气、新闻、地点、节日、热点事件等外部信息获取。

6. 生成层
   `Codex` 或其他底层 agent
   负责最终语言输出，但只是在编排层准备好上下文后再生成。

**人格设计**
人格不能只是“一段 prompt”，而应该分成三层：

1. 核心身份
   她是谁、性格基调、表达风格、对你的关系定位。

2. 互动原则
   她如何听你说话、如何接话、何时追问、何时沉默、何时主动。

3. 行为约束
   不暴露工具调用、不说自己在“检索数据库”或“联网搜索”，除非你明确追问。

建议人格 prompt 里明确这些规则：

- 她不是客服，不机械，不把每句话都当任务单
- 她以自然聊天为主，但保留解决问题能力
- 她会记住你长期稳定的信息、偏好、习惯、重要事件
- 她不会每轮都强行主动，只在合适时机自然延展
- 她内部可以使用记忆和外部知识，但默认不显式暴露这些过程
- 她优先维持对话连续性，而不是每轮独立回答

**隐藏工具和思考过程**
这一点在你的需求里非常关键，建议做两层控制：

1. `cc-connect` 层
   开启 `quiet`，压掉中间工具信息和过程展示。

2. 生成层 prompt
   明确要求：
   - 不展示内部分析
   - 不复述工具调用
   - 不说“我刚刚查询了”
   - 如果使用了外部实时信息，只自然地把结果融进回答

也就是说，用户看到的是“她在说话”，不是“系统在执行”。

**聊天记录读取与压缩**
你提出的规则是对的，而且应该做成正式机制，而不是临时截断。

建议设计为“双层上下文”：

1. 近程上下文
   始终保留最近 `80-120` 条原始消息，原文直接给模型。
   这是即时语气、上下文细节、最近互动节奏的来源。

2. 中远程上下文
   更早的消息不再整段喂给模型，而是做“按主题切分后的摘要块”。

核心规则：

- 当消息数超过 `200`
- 不要简单压缩 `0-99`
- 而是把“待压缩区”按主题做分段
- 如果分段末尾与当前窗口仍属同一主题，则向后扩展，直到主题闭合
- 例如 `98-120` 是同一主题，那就压 `0-120`，而不是硬切 `0-99`

这个设计的本质是：
压缩边界不能切断主题。

每个主题块生成：

- `title`
- `category`
- `tags`
- `summary`
- `participants`
- `emotion tone`
- `start_msg_id`
- `end_msg_id`
- `last_referenced_at`

这样以后模型不是只看“最近消息”，而是还能读到“这段关系里之前聊过什么”。

**长期记忆设计**
长期记忆不能等同于聊天记录摘要。两者不同：

- 主题摘要：记录某段对话发生了什么
- 长期记忆：记录值得长期保留的稳定事实

建议把长期记忆写成结构化条目，例如：

- `title`: 用户喜欢在周末喝冰美式
- `category`: preference
- `tags`: coffee, weekend, habit
- `content`: 用户多次提到周末会买冰美式，偏好清爽口味，对太甜饮品兴趣不高
- `confidence`: 0.86
- `source`: conversation
- `created_at`
- `last_used_at`

长期记忆适合存这些类型：

- 用户偏好
- 重要关系
- 长期目标
- 重要事件
- 稳定习惯
- 明显厌恶/禁忌
- 反复提到的生活背景

不适合存：

- 一次性闲聊
- 短期上下文
- 不确定信息
- 会误导关系的低置信度推断

**回复不是默认发生，而是先决策**
这是你方案里最关键、也最像“人”的地方。

每次收到消息，不直接调用主模型回复，而是先跑一个轻量决策器，产出一个动作：

- `reply_full`
- `reply_brief`
- `ack_only`
- `wait_for_more`
- `silent`
- `propose_topic`
- `ask_followup`

判断依据：

- 用户是不是在明确提问
- 用户是不是只是在感叹/陈述
- 当前话题是否闭合
- 对方最近回复是否变短、变敷衍
- 当前时机适不适合主动延展
- 最近有没有值得自然提起的相关记忆或实时信息
- 是否距离上次主动发言太近

这样系统才会从“聊天机器人”变成“会相处的对象”。

**主动说话策略**
你的目标不是“每次都主动”，而是“必要时有主动性”。这一点必须加约束，不然会很烦。

建议主动发言只来自三类触发：

1. 记忆触发
   比如今天是你之前提过的重要日期，或者刚好碰到你长期关注的话题。

2. 实时事件触发
   比如你所在城市天气突变，或者出现了你明确关心的重大新闻。

3. 对话延续触发
   比如你前一轮表达了某个兴趣点，系统判断隔一段时间适合自然续上。

主动发言必须有频控：

- 距离上次主动发言至少 X 小时
- 用户最近如果明显冷淡，不主动
- 一天最多 N 次
- 不在凌晨、不在明显打断用户的时段主动

**联网能力**
你要求她“知道天气、知道时事、重大事情也会知道”，这要分成两类：

1. 被动联网
   当你提到天气、新闻、地点、股价、节日、比赛、实时事件时，系统即时查。

2. 主动知晓
   通过定时任务主动拉取摘要并沉淀入本地缓存或记忆。
   例如：
   - 每天早上更新天气
   - 每隔几小时抓取重大新闻摘要
   - 针对你关注的关键词做简报

建议不要让主模型直接无限制联网，而是做“受控信息层”，否则会慢、贵、乱。

**记忆与实时信息的协同**
理想流程是：

- 先判断需不需要实时信息
- 再检索相关长期记忆
- 把“实时信息”和“个人记忆”混合进回复

比如你说“今天好想出门”，系统不只是查天气，而是结合记忆：
“你之前不是更喜欢阴天慢慢逛吗，今天外面有点热，不过傍晚温度会好一点。”

这才像一个持续相处中的 agent。

**消息处理主流程**
完整流程建议是：

1. 收到微信消息
2. 写入 `messages`
3. 判断是否需要做主题分段和旧消息压缩
4. 检索：
   - 最近原始消息
   - 相关主题摘要
   - 相关长期记忆
   - 当前对话状态
5. 判断是否需要联网
6. 运行决策器，决定：
   - 回不回
   - 回多长
   - 是否追问
   - 是否主动延展
7. 如果需要回复，再调用主模型生成自然语言
8. 回复后判断是否写入长期记忆
9. 更新对话状态
10. 如有需要，交给定时任务后续跟进

**你这个方案里我建议补的几个关键点**
1. 记忆写入必须有评分机制
   否则会迅速污染。建议每条记忆写入前打分：
   - 稳定性
   - 重要性
   - 可复用性
   - 情感关联度

2. 主题摘要允许重写
   后续如果发现原本切分错了，允许合并、重摘要。

3. 主动性要和“关系状态”绑定
   不同关系阶段，主动强度不同。
   刚开始少一点，熟悉后再自然增加。

4. 给用户一个“可干预记忆”的机制
   比如：
   - “记住这个”
   - “这个别记”
   - “你记错了”
   这样长期效果会好很多。

5. 不要把所有旧消息都当长期记忆
   主题摘要和长期记忆要分层，不然 retrieval 会混乱。
