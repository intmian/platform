# Library Independent Module Design

Date: 2026-05-23

## Summary

Library will become an independent top-level module. It will no longer run inside todone, will not use todone `Task`, `SubGroup`, `TaskSequence`, or `Task.Note`, and will not keep runtime compatibility with old `/todone/:group` library pages.

The new module uses a memory-first backend service. D1 is the durable remote store, but not the hot read path. User writes update memory immediately after a local durable outbox record is written; D1 synchronization runs in the background in user-order.

## Goals

1. Improve library performance by removing the todone `getSubGroup -> getTasks(ContainDone=true) -> parse all Task.Note JSON -> scan all logs` path.
2. Decouple library domain behavior from todone route, data model, sequence logic, and permission boundary.
3. Support slow D1 access by making reads memory-only after each user's store is loaded.
4. Preserve existing library data through a one-time migration from old todone library groups.
5. Keep writes responsive without losing accepted operations if the backend process exits before D1 flush completes.

## Non-Goals

1. No runtime compatibility layer for old todone library data.
2. No distributed multi-instance cache coherence.
3. No continued creation of todone groups with type `Library`.
4. No use of todone `Task.Note` as a shadow source after migration.

## Architecture

### Backend Boundary

Add a new backend service named `library` with RPC path:

```text
POST /service/library/:cmd
```

The service owns:

1. Library D1 tables and D1 connection configuration.
2. Per-user in-memory stores.
3. Local durable outbox for pending D1 writes.
4. Migration from old todone library groups.
5. Reload and repair operations.

The platform service registry adds `NameLibrary` and `FlagLibrary`. Library RPC accepts users with `admin` or `library` permission and requires request `UserID` to match the authenticated user, matching the existing service-level user validation pattern.

Library registers its own D1 configuration keys:

```text
library/db/account_id
library/db/api_token
library/db/db_id
```

It does not reuse todone D1 configuration keys.

### Frontend Boundary

Add a top-level frontend route:

```text
/library
```

The page lives under `frontend/src/library/`. It can reuse existing library UI concepts, but it must remove todone-specific dependencies:

1. No `Addr`.
2. No `GroupID` or `SubGroupID`.
3. No todone RPC client.
4. No `PTask` conversion.
5. No `Task.Note` parsing in the hot list path.

The app navigation adds `娱乐库` as a first-class business entry.

The old `/todone/:group` path, when it decodes a `GroupType.Library` group, displays a static migration notice and a button/link to `/library`. It does not render the old library component and does not load old library data.

The new URL state uses library-native query keys:

```text
category
statuses
todo_reason
sort
search
detail
```

## Data Model

The D1 model is optimized for load-and-replay, not frequent query joins.

`status` is the current explicit status. The derived display value `鸽了` is not stored as a permanent status. It is calculated from:

### `library_logs`

This table stores immutable or editable event records for details, timeline, score history, notes, status history, and cutoff events.


The table also stores timeline display denormalization so global timeline rebuild does not require D1 joins:

```text

```

These denormalized fields are refreshed by library service write operations and repair operations.

### IDs

Entity IDs are generated by the backend service before D1 writes, using ULID strings. D1 autoincrement IDs are not used for entities created through async write flow because the API returns before D1 accepts the write.

### D1 Indexes

Indexes support cold load, repair, and direct maintenance commands. They are not the normal interaction read path after a user store has loaded.

```text
library_items(user_id, deleted, updated_at)
library_items(user_id, deleted, category)
library_items(user_id, deleted, status, updated_at)
library_items(user_id, legacy_source, legacy_group_id, legacy_task_id)
library_rounds(user_id, item_id, index)
library_logs(user_id, item_id, time)
library_logs(user_id, time)
library_logs(user_id, type, time)
```

## Memory Store

Library service maintains a per-user store:

```text
LibraryService
  userStores: map[userID]*LibraryStore

LibraryStore
  mutex
  loadedAt
  itemsByID
  roundsByItemID
  logsByItemID
  listSnapshots
  timelineEntries
  scoreEntries
  categoryCounts
  todoReasonOptions
  syncStatus
```

The first request for a user loads that user's store. Concurrent first requests share the same load operation.

Load sequence:

```text
load items from D1
load rounds from D1
load logs from D1
load pending outbox mutations from local storage
replay pending mutations into memory
rebuild derived indexes
publish store
resume background D1 flush
```

If any step before publish fails, no partial store is published.

All hot reads use memory only:

1. List filtering.
2. Default sorting.
3. Search.
4. Category counts.
5. Todo reason options.
6. Detail retrieval.
7. Timeline.
8. Score gradient.

Personal library scale is expected to fit memory. In-memory normalized string scan is preferred over D1 search round trips.

## Write Model

Writes return after the mutation is durably recorded in local outbox and applied to memory. They do not wait for D1.

Write sequence:

```text
lock user store
validate current memory state
build mutation with service-generated IDs and next user sequence
persist mutation to local durable outbox
apply mutation to memory entities
rebuild affected derived indexes
enqueue background flush
return success
unlock
```

If local outbox write fails, the request fails and memory is unchanged.

The outbox is stored in platform-local durable storage, backed by local sqlite or equivalent `xstorage` persistence. Each record includes:

```text
mutation_id
user_id
seq
type
payload_json
created_at
attempt_count
last_error
```

D1 flush is strictly ordered per user by `seq`.

D1 operations are idempotent:

1. Item and round mutations use deterministic IDs and upsert.
2. Log creation uses deterministic IDs and insert-or-ignore or upsert.
3. Deletes are soft deletes or deterministic update operations.
4. Summary writes use the final snapshot in mutation payload so retry does not change semantics.

Flush success removes the outbox record. Flush failure keeps the record, records `last_error`, increments `attempt_count`, and retries with backoff.

## Sync Status

The service exposes `getSyncStatus`:

```text
pending_count
last_flush_at
last_error
syncing
```

The frontend displays sync state when there are pending writes or a recent flush error. D1 failures do not block reads and do not roll back accepted memory changes.

Returns:

```text

```

Items are list summaries and do not include complete logs.

### `getItem`

Returns one item with rounds and logs from memory.

### Write Commands

Write commands return the updated item snapshot or affected list/detail/timeline snapshot needed by the frontend to update without an immediate refetch.

### Admin Commands

`reloadUserStore` discards and rebuilds a user's memory store from D1 plus pending outbox replay. It must not lose pending local mutations.

`repairUserStore` flushes pending outbox first, then rebuilds item summaries and denormalized timeline fields from rounds/logs, writes repaired data to D1, and rebuilds memory.

`importFromTodoneLibrary` migrates old todone library data. It is admin-only.

## Migration

Old todone library data is migrated once. New library runtime never reads old todone data.

Migration input identifies an old todone library group. The importer reads `_library_items_` when present, otherwise the legacy first subgroup. It loads tasks and parses `Task.Note` as old `LibraryExtra`.

Conversion rules:

1. Task title becomes item title.
2. `extra.author`, `category`, `year`, `remark`, favorite flag, and cover URLs become item fields.
3. `extra.rounds` become `library_rounds`.
4. Round logs become `library_logs`.
5. Current status, status reason, status time, main score, updated time, and list summary fields are rebuilt from logs.
6. Deprecated fields are used only as fallback when logs are missing.
7. New records store `legacy_source=todone`, `legacy_group_id`, and `legacy_task_id`.

Migration is idempotent. Re-importing the same legacy source updates or skips the existing target item and does not duplicate data.

Migration writes D1 directly as an admin batch, then builds/reloads the memory store and runs repair. It does not use normal user async outbox.

Old todone data is not deleted. It remains as a backup.

Migration report returns:

```text
imported_count
skipped_count
failed_count
failed_items
```

Each failed item includes the legacy task ID and reason.

After migration, todone group creation no longer offers `Library` group type.

## Error Handling

1. User validation failure returns `no permission` or `user err`.
2. Store load failure returns a load error and does not publish partial memory.
3. Local outbox write failure fails the write request and keeps memory unchanged.
4. D1 flush failure is visible through sync status and logs, but accepted writes remain visible from memory and pending outbox replay.
5. Reload always replays pending outbox after loading D1.
6. Repair refuses to run until pending outbox is flushed successfully.
7. Outbox backlog beyond a configured threshold blocks new writes with a clear sync backlog error while reads remain available.

## Single-Instance Assumption

This design targets the platform's current single backend instance model. Memory is the authoritative hot read state for that process.

If the platform moves to multiple backend instances, this design must be extended with cross-instance invalidation, user-sticky routing, or another distributed consistency mechanism. Multi-instance coherence is not part of this design.

## Testing And Verification

### Backend Tests

1. `LibraryStore` list filtering, search, category counts, todo reason options, default sort, status derivation, wait-expired derivation, timeline, and score gradient.
2. Outbox acceptance: mutation is persisted locally, applied to memory, returned immediately, then flushed to D1 by the background writer.
3. Outbox failure: local outbox failure prevents success and leaves memory unchanged.
4. D1 flush retry: failed flush retains pending record, records error, retries, then removes record after success.
5. Crash recovery: load D1, replay pending outbox, rebuild memory, and match the pre-crash accepted state.
6. Migration conversion from representative todone `Task.Note` fixtures.
7. Migration idempotency using `legacy_group_id` and `legacy_task_id`.
8. Reload with pending outbox replay.
9. Repair refusal when pending outbox is not flushed.

### Frontend Tests

1. `/library` opens independently after login and does not need todone route state.
2. List filters, sort, search, URL query restore, detail drawer, timeline drawer, and score gradient match current library behavior.
3. Create/edit/delete/log/round operations update UI immediately from returned memory snapshot.
4. Sync status appears when pending writes or flush errors exist.
5. Old `/todone/:group` library route shows migration notice and `/library` entry.
6. Normal todone task board still loads and edits ordinary tasks.

### Performance Acceptance

1. After user store load, list filter/sort/search/detail/timeline interactions do not issue D1 queries.
2. D1 artificial delay or flush failure does not block already-loaded reads.
3. Write request latency does not include D1 round trip; it includes validation, local outbox write, memory update, and response serialization.
4. Sync status reports pending count, recent error, and last successful flush.

## Rollback

Old todone data is left untouched during migration. If the new module is disabled before users create new post-migration data, the old todone library data remains available for manual inspection or temporary restoration.

After users create data in the new module, rollback requires exporting from new library D1/outbox state; direct automatic rollback into old `Task.Note` is not part of this design.
